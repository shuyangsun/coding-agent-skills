# Export-raw-transcript v2 metadata — corrected turn counts and archive migration

## User

For raw Claude transcripts metadata generated by the export-raw-transcript skill, user turns and assistant turns are overcounted. You can find an example in ~/nas/database/llm_transcripts/raw/20260627/20260627T151717Z-claude-transcript-metadata-parser-metadata.json. I for sure did not have 75 conversations with Claude in ~/nas/database/llm_transcripts/raw/20260627/20260627T151717Z-claude-transcript-metadata-parser.jsonl.

I want user turns to represent my number of input turns, and assistant turns to represent the main agent's turn count. These numbers generated by the existing accounting method may still be useful, keep them if they are and make the JSON field name explicit about what they are.

## Assistant

Read the example metadata sidecar and `parse-claude-transcript.sh`. Found the bug at the parser's emit step: it counted **raw JSONL records by `type`** — `user_turns = count(type=="user")`, `assistant_turns = count(type=="assistant")`. In a Claude transcript every tool result is stored as a `type:"user"` record, and one assistant API response is logged across several records (a text block, then one per `tool_use`, plus any sub-agents), so both counts balloon far above real turns.

Inspected the example JSONL's structure (counts only, never content) to design accurate metrics:

- **75 `user` records** = 69 tool-results + 3 injected (`isMeta`) + **3 genuine typed messages**.
- **187 `assistant` records** collapsed to **67 distinct `message.id`** (all main-agent, 0 sidechain, 0 synthetic) — streaming/tool-use splits inflated the raw count ~2.8×.

Rewrote the parser's loop and emit:

- `user_turns` → **human input turns**: a `type:"user"` record that is not a sub-agent (`isSidechain`), not injected (`isMeta`/`isCompactSummary`/`isVisibleInTranscript==false`), and not a tool-result.
- `assistant_turns` → **main-agent turns**: distinct assistant `message.id`, excluding sidechain and synthetic (`model=="<synthetic>"`) records; an id-less record falls back to a per-record key so it still counts once.
- Kept the old raw numbers under explicit names: `user_records` / `assistant_records`.

Bumped `schema_version` 1 → 2 (the meaning of the two existing fields changed, so consumers must be able to tell v1 raw-counts from v2 turns), added the two new fields to `metadata.schema.json` with descriptions, updated `export-raw-transcript.sh` (emit the new fields, clearer `--detect` preview), and updated `SKILL.md`.

Verified against: the real example transcript (→ `user_turns=3`, `assistant_turns=67`, `user_records=75`, `assistant_records=187`); a 12-record synthetic fixture exercising sub-agents, streaming-split collapse, id-less/synthetic assistant records, and compact-summary/list-content user messages (all exact); schema validation (a valid v2 passes; old `schema_version:1` and a field-missing v2 correctly rejected); and a full live export that self-validated. My own live session at that point read **1 input / 25 agent** turns where the old method would have said 34 user / 79 assistant.
