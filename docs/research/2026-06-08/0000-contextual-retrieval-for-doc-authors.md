# Research: Contextual Retrieval Techniques for Doc Authors

**Date:** 2026-06-08
**Status:** Applied to [`updating-docs`](../../../.agents/skills/updating-docs/SKILL.md)
**Area:** context retrieval, RAG, documentation authoring
**Sources:** [Anthropic Contextual Retrieval](https://www.anthropic.com/engineering/contextual-retrieval), [Late Chunking](https://arxiv.org/abs/2409.04701), [Reconstructing Context](https://arxiv.org/abs/2504.19754), [RAPTOR](https://arxiv.org/abs/2401.18059), [GraphRAG](https://arxiv.org/abs/2404.16130), [LightRAG](https://arxiv.org/abs/2410.05779), [LongRAG](https://arxiv.org/abs/2406.15319)

## Summary

Modern retrieval systems are better at recovering context than fixed chunking, but
they still benefit from docs that state context explicitly. The practical writing
rule is: put a compact context capsule near the top, make sections self-contained,
name exact source paths and identifiers, and write relationship edges in prose and
links.

## Findings

Anthropic's contextual retrieval prepends a short chunk-specific explanation before
embedding and BM25 indexing. Their reported top-20 retrieval failure-rate reductions
were 35% for contextual embeddings, 49% for contextual embeddings plus BM25, and
67% when reranking was added. For authors, the lesson is not "let the indexing LLM
fix the doc"; it is "give the contextualizer clear parent/module/source facts to
work with."

Late chunking embeds the long document first, then pools token embeddings into
chunks. That preserves document-wide signal without LLM-generated context, but it
still depends on coherent parent documents and clean boundaries. A vague parent doc
with buried facts gives late chunking better context than naive chunking, but still
weak context.

A 2025 comparison of contextual retrieval and late chunking found the tradeoff
straight: contextual retrieval preserves semantic coherence better, while late
chunking is cheaper and can lose relevance/completeness. The doc-authoring target
should therefore support both: enough context and lexical anchors for contextual
retrieval, clean parent/section structure for late chunking.

RAPTOR, GraphRAG, LightRAG, and LongRAG all move beyond isolated small chunks:
hierarchical summaries, graph entities/relations, and larger retrieval units. These
systems reward docs that explicitly say what a section implements, depends on,
tests, supersedes, or explains. Links and source maps are not just human navigation;
they are graph edges and routing hints.

Rerankers compare the query against candidate chunks more richly than first-stage
retrieval. They help most when the candidate text contains the exact API, command,
path, error string, task ID, benchmark ID, or model/version the user means. If a
candidate says only "this change fixed it," a reranker has little to score.

## Authoring Rules

- Add a short top-of-file context capsule: date/status/area/sources plus a summary
  that names the parent project, module, workflow, and primary source paths.
- Make the first sentence of each section self-contained. Repeat the subject
  identifier at section boundaries instead of relying on pronouns.
- Mirror table-only facts in prose, especially numbers, commands, IDs, and error
  strings.
- Use source maps for cross-cutting code context. A source map should answer the
  question, name the primary code paths/functions, and link readers to verification
  sources.
- Write relationship edges explicitly: "implements", "tests", "supersedes",
  "depends on", "replaces", "generated by", "validated by".
- Keep context compact. Repeating a boilerplate paragraph in every chunk bloats the
  corpus and hurts reranking; use a source map, glossary, or split file instead.

## Implication for `updating-docs`

The skill should keep its existing "search first, update in place, front-load,
anchor lexically, link it" spine, but add one retrieval-specific rule: write the
context that advanced retrieval would otherwise have to infer. This means adding
source maps when facts span code, context lines for ambiguous docs, self-contained
sections, and relationship edges.
